# 🎨 UI/UX 实现方案：OpenClaw 项目化工作台

## UX 分析
- **项目切换必须足够明确**：当前 OpenClaw 从“单工作区”升级到“多项目”，如果项目上下文不够醒目，用户很容易误以为自己还在同一个空间里工作，导致 Chat、Agent、文件操作产生认知混乱。
- **并发状态需要“可见但不打扰”**：用户需要知道不同项目、不同 Chat 正在同时运行，但如果状态提示过多，会打断主任务流，因此运行信息应集中在顶部状态区和右侧辅助面板。
- **Chat / Session / Agent 三者必须分层展示**：主界面应优先展示用户最关心的 Chat；Session 和 Agent 应作为上下文辅助信息展示，避免把系统内部概念直接压给用户。

## 设计决策
- **配色**：整体采用深浅两套可兼容的现代工作台风格，主色建议使用偏理性科技感的蓝色系（如 `#3B82F6` / `#2563EB`），搭配中性色灰阶作为背景和分隔，强调“专业、系统、可控”的产品气质。
- **布局**：采用经典三栏结构——顶部全局栏 + 左侧项目导航 + 中间主工作区 + 右侧上下文面板。这样既符合现代 SaaS 工作台习惯，也适合承载并发状态、项目上下文和多类型内容。
- **交互**：项目切换器做成一级入口；Chat 作为主任务流；Agent、Session、Tasks 通过轻量附属面板承载。重点状态采用颜色 + 图标双表达，避免只靠颜色传递信息。
- **无障碍**：所有导航项、按钮、状态标签都应保证最小可点击区域；对比度至少达到 WCAG AA；图标按钮必须有 `aria-label`；运行状态和错误状态需要有文字辅助说明。

## 1. 设计目标

本次界面设计要解决的核心问题不是“把页面画得更漂亮”，而是：

1. 让用户始终知道自己当前在哪个项目中
2. 让项目内的 Chat、Agent、Memory、Files、Activity 结构清晰
3. 让多项目、多 Chat 并发工作时依然不混乱
4. 保留现有聊天式交互的熟悉感，同时提升系统化管理能力

---

## 2. 整体设计理念

### 2.1 关键词

- 清晰边界
- 专业工作台
- 轻量并发
- 多上下文可控
- 对工程实现友好

### 2.2 体验基调

整体体验应该更像：

- Linear / Notion / Cursor / GitHub Projects 这类“专业工具型产品”

而不是：

- 单纯聊天 App
- 层级混乱的后台系统
- 过度装饰的营销类界面

### 2.3 视觉方向

建议视觉采用：

- **顶部栏**：轻量、稳定、全局感强
- **左侧栏**：高频导航，突出项目结构
- **主内容区**：聚焦任务，尽量干净
- **右侧面板**：上下文说明，不喧宾夺主

```text
OpenClaw App
 ├── Top Bar
 │    ├── Project Switcher
 │    ├── Search

 │    └── User Menu
 ├── Sidebar
 │    ├── Chats
 │    ├── Agents
 │    ├── Files
 │    ├── Memory
 │    ├── Activity
 │    └── Settings
 ├── Main Workspace
 │    ├── Project Dashboard
 │    ├── Chat Detail
 │    ├── Agent Library
 │    ├── Files View
 │    ├── Memory View
 │    └── Activity Feed
 └── Context Panel
      ├── Session Info
      ├── Agent Info
      └── Background Tasks
```

### 核心原则

- **Project 是一级心智模型**
- **Chat 是主任务入口**
- **Agent 是角色资源，不是页面中心**
- **Session 是技术运行态，只在需要时展示**

## 4. 关键页面设计

## 4.1 顶部 Top Bar

### 包含模块

- Project Switcher
- 搜索
- 全局运行状态
- 用户菜单

### 设计重点

- 高度控制在 56~64px
- Project Switcher 靠左并足够醒目
- 搜索框视觉弱于项目切换器
- 全局状态靠右，以轻量胶囊 / 统计组件展示

### 价值

- 明确全局上下文
- 支持跨项目感知系统状态
- 不侵入主工作区

---

## 4.2 左侧 Sidebar

### 导航层级

- Chats
- Agents
- Files
- Memory
- Activity
- Settings

### 设计重点

- 当前项目名称固定在顶部
- Chat 列表支持状态标识
- 新建 Chat 按钮固定在高可见位置
- 支持折叠 Sidebar，给主内容区更多空间

### 推荐交互

- 点击导航切页面
- 点击 Chat Item 进入会话
- 当前选中项有高对比高亮态

---

## 4.3 项目 Dashboard

### 页面目标

作为项目首页，快速回答 4 个问题：

1. 最近在做什么
2. 哪些 Chat 活跃
3. 哪些 Agent 常用
4. 当前有没有后台任务

### 页面模块

- 最近 Chats
- 最近 Agents
- 最近 Activity
- 快捷操作区

### 设计重点

- 三栏或 2+1 卡片布局
- 卡片采用统一圆角和轻阴影层级
- 快捷操作按钮优先展示高频入口

---

## 4.4 Chat 详情页

### 页面目标

保留“聊天式工作流”，同时补足项目和运行态上下文。

### 页面结构

- 顶部：项目路径、Chat 标题、当前 Agent、模型、目录
- 中部：消息流
- 底部：输入区
- 右侧：Session / Agent / Tasks 面板

### 设计重点

- 消息流应是绝对主视觉中心
- 系统消息、Agent 输出、用户消息要有清晰样式区分
- 输入区固定底部，支持长文本、多行输入
- 右侧面板默认展开但可折叠

### 状态设计

- 空状态：引导用户开始该项目下首个任务
- 运行中：输入框上方可显示进度条 / “正在处理”提示
- 错误：消息区内联展示错误原因和重试按钮

---

## 4.5 Agents 页面

### 页面目标

让 Agent 成为项目内可管理资源，而不是隐藏配置。

### 页面结构

- 顶部：页面标题 + 新建 Agent 按钮
- 中部：Agent 卡片列表 / 表格
- 右侧或弹窗：Agent 详情与编辑

### 设计重点

- 每个 Agent 应至少展示：名称、角色、Skills、最近使用时间、状态
- 允许一键复制和设为默认 Agent
- 提供“从全局模板导入”入口

---

## 4.6 Files / Memory / Activity 页面

### Files
用于管理项目 workspace 和目录边界，应偏结构化、偏工具感。

### Memory
用于展示项目长期记忆和 Chat 摘要，应偏知识面板风格。

### Activity
用于查看项目最近发生的行为，应偏事件流 / 审计流风格。

### 共同设计要求

- 明确区分“查看”和“操作”
- 支持搜索 / 筛选 / 排序
- 视觉上统一为同一套工作台子页面

---

## 5. 关键组件设计建议

### 5.1 Project Switcher

建议样式：

- 胶囊按钮 + 下拉箭头
- 当前项目名突出
- 下拉中显示最近项目和管理入口

### 5.2 Chat Item

建议包含：

- 标题
- 默认 Agent
- 最近更新时间
- 状态点 / 状态图标
- 可选的后台任务标记

### 5.3 Status Badge

统一状态系统：

- Active：蓝色
- Running：绿色 / 蓝绿色
- Waiting：黄色
- Error：红色
- Archived：灰色

并保证有文字标签，不只依赖颜色。

### 5.4 Context Panel Card

右侧上下文面板建议使用小卡片：

- Session Info
- Agent Info
- Background Tasks
- Recent Logs

这样更便于按需扩展。

## 6. 响应式策略

### Desktop
- 三栏布局完整显示
- Sidebar 默认展开
- Right Panel 默认可见

### Tablet
- Sidebar 可折叠
- Right Panel 改抽屉式
- 主内容优先

### Mobile
- 顶部保留 Project Switcher
- Sidebar 改侧滑抽屉
- Right Panel 收纳到 Tab / Bottom Sheet
- Chat 输入区固定底部

### 设计建议

除非明确要求桌面专用，否则所有页面设计应考虑移动端层级压缩。

## 7. 可用性与无障碍优化点

### 尼尔森原则对应优化

1. **系统状态可见性**
   - 顶部全局运行状态
   - Chat 内运行进度与错误提示

2. **系统与现实世界匹配**
   - 用“项目、聊天、代理、活动”这些自然语言，不要堆技术术语

3. **用户可控性**
   - 允许停止后台任务
   - 允许切换 Agent / 切换项目

4. **一致性与标准**
   - 所有状态标签、弹窗、卡片使用统一样式语言

5. **错误预防**
   - 文件写操作或跨项目访问需要提示与确认

### WCAG 重点

- 主文本与背景对比度满足 AA
- 图标按钮必须有文本提示或 `aria-label`
- 点击区域建议不小于 40x40
- 错误提示不能只靠红色区分
- 焦点态可见

## 8. 视觉风格建议

### 色彩系统

- Primary：蓝色系（科技、稳定、专业）
- Neutral：冷灰色系
- Success：绿色
- Warning：琥珀黄
- Error：红色

### 字体层级

- 页面标题：20~24px / 600
- 模块标题：16~18px / 600
- 正文：14~16px / 400
- 辅助信息：12~13px / 400

### 间距系统

建议采用 4 / 8 / 12 / 16 / 24 / 32 的 spacing scale。

### 圆角与阴影

- 卡片圆角：12~16px
- 弹窗圆角：16~20px
- 阴影轻量，不走过度拟物风
